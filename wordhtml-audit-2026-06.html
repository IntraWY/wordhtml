สรุปผู้บริหาร (Executive Summary)

wordhtml เป็นแอป client-side ที่โครงสร้างแข็งแรงและ feature ครบตามแผนเดิมแล้ว — ทั้ง 17 ข้อจาก feature proposal ถูก ship ครบ จุดที่ต้องลงทุนต่อคือ ความเสถียร (test coverage ของ pagination/components ยังต่ำ), งานครึ่งทาง (header/footer, table split ที่มีโค้ดอยู่แล้วแต่ยังไม่ wire), และ การ harden ระบบ placeholder ให้ปลอดภัยต่อการใช้งานจริงระดับเอกสารราชการ

จุดแข็ง vs ความเสี่ยง

✅ จุดแข็ง (Strengths)

  • Export ครบ 5 รูปแบบ (HTML / DOCX / MD / PDF / ZIP) + Mail-merge + Distribution list
  • ฟีเจอร์เอกสารราชการไทยครบ: หนังสือภายนอก, บันทึกข้อความ, บล็อกลงนาม, เลขไทย, บาทเท็กซ์
  • ระบบ Placeholder/Variable มี test coverage สูงสุดในโปรเจค (95%)
  • Cloud history sync + Draft recovery สองชั้น (local 8s + snapshot 2 นาที)
  • Pagination แบบ holistic reflow พร้อม soft-split ข้ามหน้า (repaginate.ts)
  • สถาปัตยกรรม client-side ล้วน — privacy story ชัดเจน ไม่มี server

⚠️ ความเสี่ยงหลัก (Key Risks)

  • Pagination เพิ่ง refactor ใหญ่แต่ test แค่ 45% — เคยมี defect reproduce ได้ 3 เคส ยังไม่มี E2E คุ้มกัน
  • มี pagination engine 2 ตัว (editor vs preview) — ผลแบ่งหน้าอาจไม่ตรงกันในเอกสารยาว
  • Filter ของ placeholder fail เงียบ ๆ เมื่อ type ไม่ตรง — เสี่ยงข้อมูลหายตอน export จริง
  • Component ใหญ่ (ExportDialog 677 บรรทัด) ไม่มี unit test — แก้แล้วพังยากจะรู้ตัว
  • a11y มีปัญหา critical: <html lang="en"> ทั้งที่เนื้อหาไทย, contrast ไม่ผ่าน WCAG AA, ruler ใช้คีย์บอร์ดไม่ได้
  • Dead code: indentExtension.ts (119 บรรทัด) ไม่ถูก register ที่ไหนเลย
02

เจาะลึกระบบ Placeholder

ระบบ placeholder คือหัวใจของโหมด template / mail-merge — ครอบคลุม merge fields {{name}}, filters {{x|baht}}, page tokens {page_th}, fill-in fields, panel สถานะ และ export health check โดยรวมเป็นระบบที่แข็งแรงที่สุดในโปรเจค แต่การตรวจเชิงลึกพบช่องว่าง 10 ข้อที่ควรปิดก่อนใช้งานจริงเข้มข้น

เส้นทางข้อมูล (Data Flow)

1 · Edit Mode เอกสารในแก้ไข — variable badge & plain tokens

ผู้ใช้พิมพ์ {{ → autocomplete (variableSuggestion.ts) → แสดงเป็น badge ผ่าน variableMark.ts พร้อม guard กันพิมพ์ทับ (variableTypingGuard.ts)

2 · Placeholder Panel (View ribbon) ตรวจสถานะ + จัดการตัวแปร

PlaceholderPanel.tsx มี 3 แท็บ: ตัวแปร (สถานะจาก getMergeFieldStatuses()), หัว/ท้ายกระดาษ (listPageTokensIn()), คำเตือน (countMissingFields()) + ปุ่ม jump ไปยังตัวแปรในเอกสาร

3 · Preview Mode แทนค่าด้วย data row + แสดงตัวที่ขาดเป็นสีแดง

replaceMergeFields(html, vars, dataRow, {mode:"preview"}) — ตัวแปรที่ไม่มีค่าจะแสดงเป็น .placeholder-missing ส่วน header/footer overlay ทำผ่าน PageChromeLayer.tsx + resolveHeaderFooter() (first/odd/even)

4 · Export Dialog inline fields → cleaners → resolve → health check

ลำดับใน ExportDialog.tsx: inlinePlaceholderFields() → applyCleaners() → resolveHtmlPlaceholders({mode:"export", missingPolicy}) → checkExportHealth() (error = เอกสารว่าง, warning = ตัวแปรขาดใน preview)

5 · Format-Specific Export HTML / DOCX / Mail-merge ZIP / Distribution ZIP

Mail-merge: buildMergedDocuments() resolve ใหม่ทีละ data row → JSZip · Distribution: buildDistributionDocuments() แทน {{เรียน}} ทีละผู้รับ — สองท่อนี้แยกกัน รวมกันไม่ได้ (gap #9)

ช่องว่าง 10 ข้อ พร้อมหลักฐานในโค้ด

GAP 01Filter ไม่ validate type — fail เงียบHIGH

ใน src/lib/placeholders/mergeFields.ts ฟังก์ชัน applyMergeFilter() ไม่ตรวจว่าค่าตรงกับ filter หรือไม่ — {{username|baht}} กับค่าที่ไม่ใช่ตัวเลขจะได้ string ว่าง โดยไม่มีคำเตือนใด ๆ (เพราะ bahtText() เช็ค Number.isFinite() แล้วคืนค่าว่าง) เช่นเดียวกับ |date ที่ parse ไม่ได้

เพิ่ม pre-validation ใน applyMergeFilter และรายงานเป็น issue ใหม่ใน checkExportHealth() (severity: warning) — Phase 2

GAP 02Fill-in fields ไม่มี UI กรอกค่าก่อน exportHIGH

src/lib/tiptap/placeholderField.ts มี node + คำสั่ง insert ครบ และ store มี fieldValues + setFieldValue() แล้ว แต่ไม่มี component ไหนให้ผู้ใช้กรอก/ตรวจค่าระหว่าง insert จนถึง export — inlinePlaceholderFields() จะแปลง field เป็น label หรือค่าว่างทันที

เพิ่มฟอร์มกรอกค่า fields ใน ExportDialog.tsx + enforce required fields ก่อนปล่อยดาวน์โหลด — Phase 2

GAP 03Filter มีแค่ 3 ตัว ทั้งที่ type ระบบรองรับมากกว่าMEDIUM

src/lib/placeholders/constants.ts ประกาศ MERGE_FIELD_FILTERS = ["baht","thai","date"] ขณะที่ src/types/template.ts มี VariableType = "text"|"number"|"currency"|"date"|"percent" — currency / percent ไม่เคยถูกใช้ และ field format, isComputed, expression ใน TemplateVariable เป็น dead schema

เพิ่ม filters: |currency |percent |comma |upper |lower — Phase 2

GAP 04ตัวแปรว่างไม่ block export (เป็นแค่ severity "info")MEDIUM

ใน src/lib/export/exportHealthCheck.ts เมื่ออยู่ template mode และมีตัวแปรไม่มีค่า ระบบ push issue ด้วย severity: "info" เท่านั้น — export เดินหน้าต่อได้ ผลคือไฟล์ที่มี [ชื่อตัวแปร] หรือช่องว่างหลุดออกไปโดยไม่รู้ตัว อันตรายเป็นพิเศษกับ mail-merge หลายสิบไฟล์

เพิ่มตัวเลือก "Strict mode" ใน export dialog — ถ้าเปิด ให้ unset variables เป็น severity "error" และ block การ export — Phase 2

GAP 05Page token ที่ไม่รู้จักกลายเป็นค่าว่างเงียบ ๆ + ไม่มี help textMEDIUM

src/lib/placeholders/pageTokens.ts — replacePageTokens() คืน "" สำหรับ token นอกรายการ และ Panel ไม่มีที่แสดงรายการ token ที่รองรับทั้งหมด ผู้ใช้ต้องเดาเอง token ที่ขาด: {time}, {filename}, {section}

เพิ่ม help text รายการ token ในแท็บ "หัว/ท้าย" ของ PlaceholderPanel + เพิ่ม token {time}, {filename} — Phase 2

GAP 06Jump-to-field fail เงียบ และไม่มี testMEDIUM

src/lib/placeholders/jumpToMergeField.ts พึ่ง search-and-replace extension — ถ้า extension หายหรือพัง ฟังก์ชันคืน false โดยผู้ใช้คลิกแล้วไม่เกิดอะไรขึ้น ไม่มี toast แจ้ง และโมดูลนี้ไม่มี unit test เลย

เพิ่ม toast แจ้งเมื่อ jump ล้มเหลว + เขียน unit test — Phase 2

GAP 07ไม่มี field-level validation / custom format / computed fieldsLOW

TemplateVariable.type ไม่ถูก enforce ตอน replaceMergeFields(), ไม่มี {{date|format:dd/mm/yyyy}}, และ expression ไม่เคยถูก evaluate — นักออกแบบ template กำหนดข้อบังคับ format ไม่ได้

Backlog — ทำหลัง strict mode + filter validation เสถียรแล้ว (Phase 4–5)

GAP 08ไม่มี conditional blocks / loopsLOW

ไม่รองรับ {{#if name}}...{{else}}...{{/if}} หรือ loop สำหรับ list variables — template ปรับเนื้อหาตามการมี/ไม่มีข้อมูลไม่ได้ (เช่น ซ่อนย่อหน้า "สิ่งที่ส่งมาด้วย" เมื่อไม่มีไฟล์แนบ)

เพิ่ม mini template syntax ใน resolve pipeline — Phase 4

GAP 09Mail-merge × Distribution รวมกันไม่ได้MEDIUM

src/lib/export/distributionList.ts hardcode แทนที่ field เดียว ({{เรียน}}) จาก templateHtml ดิบ — ไม่เชื่อมกับ dataSet ของ mail-merge เลย จึงสร้างจดหมาย personalized ถึงผู้รับหลายคนจากชุดข้อมูลเดียวไม่ได้ (1 ไฟล์ต่อคู่ row×recipient)

รวมท่อ: resolve ตัวแปรจาก data row ก่อน แล้วค่อย distribute ต่อผู้รับ — Phase 2

GAP 10โมดูลใน critical path ที่ไม่มี test เลย 4 ตัวHIGH

โมดูลที่อยู่ในเส้นทาง export/UI แต่ coverage = 0%: jumpToMergeField.ts · inlinePlaceholderFields.ts (อยู่ในท่อ export — ถ้า selector พัง ข้อมูลหายเงียบ) · placeholderField.ts · exportHeaderFooter.ts (generate CSS ซับซ้อนโดยไม่มีการ verify)

เขียน unit tests ครบ 4 โมดูล — Phase 2 (ทำก่อนแตะ logic ใด ๆ ในนั้น)

Roadmap การพัฒนา 5 เฟส
